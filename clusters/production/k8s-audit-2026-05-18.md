# Kubernetes GitOps Environment - Full Audit Report

**Date:** May 2026
**Scope:** ~210 YAML files across `/home/wrees/k8s-gitops`
**Platform:** Flux CD v2.7.5, bare-metal Kubernetes cluster, Cilium CNI + Gateway API
**Previous audit:** `k8s-audit-2026-02-18.md`

This is a follow-up to the February 2026 audit. It records the substantial
progress made since then, remediates the outstanding and newly-found issues in
a series of commits, and lists the items that still need a decision or live
testing before they can be actioned.

## Remediated in This Audit

The following were fixed in this branch (`audit-2026-05-remediation`), one
commit per item:

### R1. NetworkPolicy coverage gaps — REMEDIATED
- **Issue:** `cert-manager`, `monitoring`, `node-feature-discovery`, and
  `intel-gpu-plugin` had application workloads but no `CiliumNetworkPolicy`.
- **Fix:** Added an `ingress-baseline` policy to each, replicating the existing
  pattern (`utilities/utilities-networkpolicy.yaml`).
- **Files:** `cert-manager/cert-manager-networkpolicy.yaml`,
  `monitoring/monitoring-networkpolicy.yaml`,
  `core/node-feature-discovery/node-feature-discovery-networkpolicy.yaml`,
  `media/intel-gpu-plugin/intel-gpu-plugin-networkpolicy.yaml`.
- **Deliberately skipped:** `kube-system` and `longhorn-system` (system
  namespaces — restrictive policies there are high-risk and Longhorn is not
  GitOps-managed); `gateway` (no application pods).

### R2. Unused istio resources — REMEDIATED
- **Issue:** `rees-namespace.yaml` defined an `istio-system` namespace and
  `sources/istio-source.yaml` defined an istio HelmRepository, but no istio
  HelmRelease exists anywhere (Feb #21).
- **Fix:** Removed the `istio-system` namespace block and deleted
  `sources/istio-source.yaml`.

### R3. metermon missing resource limits — REMEDIATED
- **File:** `automation/metermon/metermon.yaml`
- **Issue:** No CPU/memory requests or limits — no QoS guarantee, noisy-neighbour risk.
- **Fix:** Added requests (`50m`/`64Mi`) and limits (`250m`/`128Mi`).

### R4. metermon implicit `:latest` image — REMEDIATED
- **File:** `automation/metermon/metermon.yaml`
- **Issue:** `image: seanauff/metermon` had no tag (resolves to `:latest`).
- **Fix:** Upstream publishes only `:latest`, so the image is now pinned by
  digest (`seanauff/metermon:latest@sha256:01264bae…`). Bump manually when an
  update is wanted.

### R5. Containers allowed privilege escalation — REMEDIATED
- **Issue:** Application containers did not set `allowPrivilegeEscalation`, so a
  setuid binary could be used to escalate inside the container.
- **Fix:** Added `securityContext.allowPrivilegeEscalation: false` to **26
  application containers** (the *arr stack, Plex, Immich, CompreFace, voice
  services, Vaultwarden, mosquitto, double-take, oscar, ps, minecraft-bettermc).
- **Not changed:** privileged / `hostNetwork` workloads (cilium, frigate,
  esphome, music-assistant, govee2mqtt, wyzebridge, metermon) and Flux's own
  `gotk-components.yaml`. `rsyncd-backups` already had a hardened context.
- **Note:** deeper hardening (`capabilities: drop: ["ALL"]`, `runAsNonRoot`) was
  **not** blanket-applied — see finding M3.

### R6. Missing health probes — REMEDIATED
- **Issue:** Only ~8 workloads had liveness/readiness probes; a hung or crashed
  container would not be restarted or removed from Service endpoints (Feb #15).
- **Fix:** Added `startupProbe` + `livenessProbe` + `readinessProbe` (TCP socket,
  generous startup grace) to **20 web/service workloads**, and an exec
  `pg_isready` readiness probe to the CompreFace database.
- **Deliberately skipped:** `minecraft-bettermc` — the AUTO_CURSEFORGE modpack
  has a highly variable first-boot time; a probe risks a restart loop without
  live tuning. Immich server/db/redis already had readiness probes.

### R7. Longhorn backup config not version-controlled — REMEDIATED
- **Issue:** Volume backups run to a CIFS target, but the `BackupTarget` and
  `RecurringJob` existed only in the cluster (created via the Longhorn UI) — not
  reproducible on a rebuild.
- **Fix:** Added `core/longhorn/longhorn-backup.yaml` codifying the live
  `default` BackupTarget (`cifs://10.255.255.79/longhorn`, `cifs-secret`
  credential) and the `c-1uij3f` RecurringJob (daily backup, retain 15). Names
  match the live resources exactly, so Flux **adopts** them — no duplicate jobs.

## Outstanding Findings (Need a Decision or Live Testing)

### HIGH

#### H1. CompreFace runs PostgreSQL 11.5 (EOL October 2023)
- **File:** `automation/compreface/1-db.yaml` — `image: postgres:11.5`
- **Issue:** PostgreSQL 11 is end-of-life; no security patches, multiple known CVEs.
- **Why not auto-fixed:** A major-version jump (11 → 16) is not an in-place image
  swap — the on-disk data directory is incompatible. Requires a `pg_dump` /
  `pg_restore` (or `pg_upgrade`) migration and a CompreFace compatibility check.
- **Recommendation:** Schedule a maintenance window: dump the `frs` database,
  deploy `postgres:16`, restore, verify CompreFace.

#### H2. music-assistant runs with SYS_ADMIN capability
- **File:** `automation/music-assistant/music-assistant-release.yaml`
- **Issue:** Adds `SYS_ADMIN` and `DAC_READ_SEARCH` capabilities. `SYS_ADMIN` is
  close to full root and is a well-known container-escape vector (Feb #9).
- **Why not auto-fixed:** Removing capabilities risks breaking the app; needs
  testing to find the genuine minimum set.
- **Recommendation:** Test with both capabilities removed; if a feature breaks,
  re-add only the specific capability required and document why.

#### H3. Broad cluster-admin access
- **Files:** `monitoring/headlamp/admin-sa.yaml`, `identity/oidc-cluster-admin.yaml`
- **Issue:** Two standing paths to full cluster control: a static ServiceAccount
  (`warren`) bound to `cluster-admin`, and the `oidc:kubernetes-admins` group
  bound to `cluster-admin`. Headlamp also authenticates users via OIDC, so the
  static token is likely redundant (Feb #4).
- **Why not auto-fixed:** Removing the static SA could lock you out of Headlamp
  if OIDC login is not fully working.
- **Recommendation:** Confirm OIDC login to Headlamp works, then delete the
  static `warren` ServiceAccount + ClusterRoleBinding. Audit who is in the
  `oidc:kubernetes-admins` group in Authentik.

### MEDIUM

#### M1. property-search uses a `:latest` tag
- **File:** `utilities/ps/ps-release.yaml` — `warrenrees/property-search:latest`
- **Issue:** Unpredictable deploys; no record of what version is running (Feb #14).
- **Why not auto-fixed:** This is your own image and the registry path is not
  publicly resolvable, so a concrete tag could not be determined.
- **Recommendation:** Tag releases of `property-search` (date or semver), pin the
  tag here, and add a Flux `ImageRepository`/`ImagePolicy` so it joins the
  automated update flow like the other workloads.

#### M2. Flux image automation commits directly to `main`
- **File:** `flux-image-updates/flux-system-automation.yaml`
- **Issue:** `ImageUpdateAutomation` pushes image-tag bumps straight to `main`
  every minute, with no review step (Feb #20).
- **Recommendation:** Point the automation at a `flux-image-updates` branch and
  open a PR, or accept the trade-off explicitly (it is reasonable for a homelab).

#### M3. Container hardening stops at `allowPrivilegeEscalation`
- **Issue:** After R5, containers still run with the default Linux capability set
  and (mostly) as root. Pod Security Standards `baseline`/`privileged` are
  enforced at the namespace level, but the `restricted` profile is not met.
- **Why not auto-fixed:** Blanket `capabilities: drop: ["ALL"]` would break
  several workloads — linuxserver.io images chown `/config` at startup (need
  `CHOWN`, `FOWNER`, `DAC_OVERRIDE`, `SETUID`, `SETGID`), Vaultwarden and the
  CompreFace frontend bind port 80 (need `NET_BIND_SERVICE`), and the Postgres
  images drop privileges via `gosu`. This needs a per-app capability allowlist
  and live testing.
- **Recommendation:** Per workload, add `capabilities: { drop: ["ALL"], add: [...] }`
  with the minimum set, and `runAsNonRoot` where the image supports it. Tackle a
  few apps at a time and verify each.

#### M4. No log aggregation
- **Issue:** Metrics (Prometheus/Grafana) are in place, but there is no central
  log store — `kubectl logs` is the only option, and logs are lost on pod restart.
- **Recommendation:** Add Loki + Promtail/Alloy to the monitoring stack; Grafana
  is already present to query it.

### LOW / Informational

#### L1. `hostNetwork: true` on 5 workloads
- esphome, music-assistant, govee2mqtt, metermon, wyzebridge. Each has a genuine
  reason (mDNS / device discovery / camera streaming). No action required beyond
  keeping the list documented; these are intentionally excluded from R5/R6.

#### L2. Privileged containers
- cilium (CNI), frigate (Coral/GPU), esphome (serial devices). Justified;
  re-confirm periodically that device-plugin alternatives are not yet viable.

#### L3. minecraft-bettermc has no probes
- Skipped in R6 by design (variable modpack boot time). Add a `startupProbe` with
  a large `failureThreshold` once a typical first-boot duration is known.

#### L4. Single-replica workloads / no PDBs for them
- Most apps run a single replica with no PDB. Acceptable for a homelab (a node
  drain causes brief downtime, not data loss; storage uses `Retain`). Listed for
  awareness, not as a defect.

## Verification Steps

After this branch is merged to `main` and Flux reconciles:

1. `flux reconcile kustomization flux-system --with-source` — force a sync.
2. `flux get kustomizations -A` / `flux get helmreleases -A` — all `Ready`.
3. `kubectl get pods -A` — no `CrashLoopBackOff` or `Pending`. Pay attention to
   the workloads that got new probes (R6) and securityContext (R5); watch for
   probe-induced restarts in the first few minutes.
4. `kubectl get ciliumnetworkpolicy -A` — the four new `ingress-baseline`
   policies are present; confirm Grafana/Prometheus/cert-manager still reachable.
5. `kubectl -n longhorn-system get backuptarget,recurringjob` — exactly **one**
   of each (`default`, `c-1uij3f`); confirm in the Longhorn UI that no duplicate
   job was created and the next backup runs on schedule.
6. `kubectl -n media describe pod <app>` — probes report healthy; no restarts
   attributable to liveness failures.
